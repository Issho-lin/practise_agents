# 语音助手实现全景梳理

> 本文档基于当前代码实际实现整理，不以 `.trae/skills/voice-control-integration/SKILL.md` 为准。
> 后续新增页面、模块、筛选项、后端 AI 通道时，应优先参考本文档与真实代码。

## 1. 文档目的

本文档回答 5 个问题：

1. 这个项目围绕语音助手已经做了哪些事情。
2. 这些能力分别是怎么执行起来的。
3. 现有实现涉及了哪些文件、每个文件负责什么。
4. 当前哪些能力已经落地，哪些仍是预留或待后端接入。
5. 后续新增模块、页面、筛选项时，应该如何接入。

---

## 2. 当前结论摘要

当前项目已经完成的是一套 **“前端本地语音控制执行骨架”**，核心链路是：

`语音输入/文本输入 -> ASR 识别 -> 本地解析 -> 执行编排 -> 页面适配器 -> 模块控件 -> UI 反馈`

当前已落地的能力包括：

- 全局语音助手浮层入口和结果面板。
- 语音识别接入能力，包括 WebSocket ASR 与浏览器原生 ASR 两套识别器。
- 语音文本的本地解析器，支持页面跳转、全屏、首页筛选、驾驶舱筛选、模块打开、模块 tab/select 切换、澄清追问。
- 页面适配器机制，已接入 `home` 和 `opcenter` 两个页面。
- 模块注册机制，已支持多个业务模块在运行时声明自己可被语音操控。
- 模块静态元数据中心，用于跨页面解析和 fallback 匹配。
- 命令执行编排器，负责动作排序、依赖、等待、跨页跳转后的继续执行。
- 弹窗与页面双实例状态同步机制，保证语音切换后弹窗里的控件也同步。
- 调试模式与语音模式双入口。

当前**还没有真正落地**的能力：

- 后端 AI 意图识别服务的正式接入。
- `DataLoom`/“问数金灵”业务 WebSocket 指令通道的正式接入。
- 基于后端返回 `action_plan` 的协议映射执行。

也就是说：

- 现在项目**已经具备前端本地语音控制能力**。
- 但“ASR final 文本 -> 后端 AI -> 返回动作计划 -> 前端执行”的这段链路，**目前还是预留状态**。

## 3. 已经做了什么

### 3.1 全局语音助手入口

项目在应用根组件里挂载了全局语音助手面板：

- `src/App.vue`
- 全局渲染 `VoiceCommandPanel`
- 非登录等页面隐藏逻辑由 `useVoiceCommand()` 控制

这意味着语音能力不是某个单页临时功能，而是被做成了整个驾驶舱的全局基础能力。

### 3.2 统一语音面板

项目已经实现了一个统一的语音交互面板：

- 文件：`src/voice/components/VoiceCommandPanel.vue`
- 功能：
  - 展示语音助手悬浮按钮
  - 展示“正在聆听 / 处理中 / 错误”状态
  - 展示实时识别结果
  - 展示完整识别结果列表
  - 展示执行结果文案
  - 对全屏操作提供二次确认
  - 支持 debug 文本输入模式

这说明项目不只是做了底层能力，也已经做了用户可见的交互界面。

### 3.3 识别引擎层

项目已做两套识别器：

1. `src/voice/recognizers/browserRecognizer.ts`
   - 浏览器原生 `SpeechRecognition`
   - 作为兼容或早期兜底实现

2. `src/voice/recognizers/cloudRecognizer.ts`
   - 基于 WebSocket 的流式识别器
   - 连接 ASR 服务
   - 建连后等待服务端 `ready`
   - 录音器持续发送 PCM 数据
   - 收到 `partial/final` 后回传文本

3. `src/voice/recognizers/audioRecorder.ts`
   - 负责麦克风采集
   - 转单声道
   - 16kHz 重采样
   - 转成 16bit PCM

4. `src/voice/recognizers/asrWebSocketConfig.ts`
   - 负责 ASR WebSocket 地址解析
   - 支持从 URL 参数、`localStorage`、全局变量覆盖
   - 本地开发时会默认回退到 `ws://localhost:6006/ws`

也就是说，项目在“语音采集与识别”这一层已经做得比较完整，不是纯 mock。

### 3.4 文本解析层

项目已实现本地解析器：

- 文件：`src/voice/core/parser.ts`
- 核心职责：
  - 从自然语言中抽取页面、月份、年度周期、组织、tab、看板类型、模块及模块控件动作
  - 根据上下文推断目标页面和目标看板
  - 生成结构化 `ParsedCommand`
  - 在候选模块冲突时生成 `clarify`
  - 在没有运行时模块注册时，仍可依赖静态模块定义做 fallback 解析

项目已配套实现：

- `src/voice/core/dictionary.ts`
  - 同义词字典
  - 页面别名
  - 全屏别名
  - Tab 别名
  - 日期、组织等正则模式
  - 看板类型映射

- `src/voice/core/modules.ts`
  - 导出静态模块 fallback 列表

- `src/voice/core/module-definitions.ts`
  - 集中维护静态模块元数据
  - 包括模块标题、别名、控件定义、静态 options
  - 提供 `hydrateModuleDef()` 将静态模块上下文水合为运行时可执行定义

### 3.5 执行编排层

项目已经实现比较完整的执行编排机制：

- `src/voice/core/dispatcher.ts`
  - 对外统一的命令分发入口
  - 注入全局导航和全屏能力
  - 收集页面上下文
  - 调用 orchestrator 执行命令

- `src/voice/core/orchestrator.ts`
  - 负责动作排序
  - 负责跨页面导航后继续执行
  - 负责动作依赖检查
  - 负责页面 ready/header ready/tabs ready 等等待机制
  - 汇总动作执行结果并生成用户提示文案

- `src/voice/registry/executionRules.ts`
  - 为不同页面定义动作执行顺序
  - 定义依赖关系、等待点、是否允许执行

也就是说，当前系统不是“匹配到动作后直接硬调用”，而是存在一层明确的编排规则。

### 3.6 页面适配器层

项目已经把不同页面的执行能力封装成适配器：

- `src/voice/adapters/homeAdapter.ts`
- `src/voice/adapters/opcenterAdapter.ts`

适配器负责：

- 判断页面是否 ready
- 读取当前页面上下文
- 暴露当前页面可选组织、Tab
- 执行页面内动作
- 返回标准化 `ActionResult`

页面本身则通过 `defineExpose` 暴露可供语音层调用的能力：

- 首页：
  - `src/views/home/index.vue`
  - `src/views/home/FilterTopBar.vue`

- 驾驶舱：
  - `src/views/opcenter/index.vue`
  - `src/views/opcenter/components/Header.vue`

### 3.7 模块注册层

项目已经支持业务模块主动注册自己可被语音操控：

### 3.8 弹窗状态同步层

项目已经处理了一个非常实际的问题：

> 页面主实例中的图表状态切换后，如果弹窗里渲染的是该组件的另一份实例，如何让弹窗也同步到最新语音操作结果？

为此新增了：

- `src/voice/store/voiceControlState.ts`
  - 以 `modalComponentId + controlId` 作为键
  - 发布控件值更新
  - 通过版本号避免旧状态覆盖新状态

业务 hook 会 watch 这个 store：

- 页面执行完模块切换后发布状态
- 弹窗组件挂载后立即回放最新状态

这部分是当前实现里非常关键、也很容易在旧文档中被遗漏的一块。

## 4. 整体架构怎么执行

## 4.1 执行总链路

```text
用户点击语音助手
  -> useVoiceControl 决定进入 debug 模式或 backend 模式
  -> CloudRecognizer / BrowserRecognizer 开始识别
  -> AudioRecorder 采集音频并推送 PCM
  -> 识别器持续返回 partial / final
  -> final 文本进入 useVoiceControl
  -> 本地 parser 结合页面上下文解析为 ParsedCommand
  -> dispatcher 识别是全局动作还是页面动作
  -> orchestrator 按 executionRules 排序执行
  -> pageAdapter 执行页面动作
  -> moduleControlRegistry 执行模块控件切换
  -> voiceControlStore 同步弹窗内重复实例
  -> VoiceCommandPanel 展示最终结果
```

## 4.2 运行上下文是怎么来的

执行前，系统会动态收集当前页面上下文：

- 当前页面 ID
- 当前看板类型
- 页面可选组织
- 页面可选 tab
- 当前已注册模块及控件

上下文来源：

- 页面适配器：`pageAdapterRegistry.getCurrent()`
- 当前页面上下文：`adapter.getCurrentContext()`
- 运行时模块：`moduleControlRegistry.getContexts()`

上下文会注入给 parser，提升以下能力：

- 在当前页面下精确识别组织
- 识别当前页面 tab
- 识别当前页面已经挂载的模块
- 从运行时模块控件里识别可切换选项

## 4.3 动作为什么能跨页面执行

跨页面时，执行逻辑不是断掉的，而是：

1. parser 先识别出目标页面与动作。
2. orchestrator 发现需要导航时，先执行 `navigate`。
3. 导航前把完整命令放入 `pendingCommandStore`。
4. 目标页面挂载后注册自己的 pageAdapter。
5. orchestrator 等待目标页面 `whenReady()`。
6. 页面 ready 后继续执行剩余动作。

这是当前语音系统最重要的基础能力之一。

## 4.4 模块为什么能跨页面解析

即使目标页面的模块还没挂载，parser 仍然可能识别它，是因为：

- 运行时优先读取 `moduleControlRegistry` 中已注册模块
- 若目标页面还没挂载，则回退到 `module-definitions.ts` 中的静态模块定义

也就是说：

- 静态模块定义用于“理解语义”
- 运行时模块注册用于“真正执行”

关键约束：

- `openModule`、`setModuleTab`、`setSelect` 均依赖 `setBoardType`

## 7. 当前有哪些输入模式

## 7.1 Debug 模式

特点：

- 不开语音识别
- 直接在面板中输入文本
- 走本地 parser 与本地执行链路

用途：

- 方便开发调试
- 不依赖麦克风
- 不依赖 ASR 服务

## 7.2 Backend 模式

特点：

- 打开语音识别
- ASR final 文本进入 backend 分支

但当前现状是：

- `requestBackendIntent()` 仍未接入真实后端
- 最终还是回退到 `createIntentFromText()`
- 所以 backend 模式目前本质是“语音输入 + 本地解析执行”

这点必须明确，否则很容易误以为项目已经打通 AI 后端。

## 9. 关键文件分别做什么

## 9.1 `useVoiceControl.ts`

这是语音系统的前端主控入口，负责：

- 初始化识别器
- 管理面板状态
- 接收 partial/final 结果
- 决定走本地分支还是 backend 分支
- 创建 parser 上下文
- 调用 dispatcher 执行
- 保存和展示 actionResult

如果以后接 DataLoom，这里会是首要改造入口。

## 9.2 `parser.ts`

这是“理解用户话”的核心：

- 识别页面
- 识别月份和年度周期
- 识别首页组织和 tab
- 识别看板类型
- 识别模块与模块控件
- 遇到冲突时返回 `clarify`

它当前仍是整个系统的语义入口核心。

## 9.3 `orchestrator.ts`

这是“怎么执行”的核心：

- 对动作排序
- 执行导航
- 等待页面 ready
- 检查动作依赖
- 调用页面适配器
- 汇总执行结果

如果没有这个文件，语音系统只能做成简单 if/else，无法稳定跨页面执行。

## 9.4 `module-definitions.ts`

这是“模块语义字典”：

- 维护模块 ID
- 维护模块标题和 aliases
- 维护静态控件结构与静态 options
- 为 parser 提供 fallback 语义支持

这是新增模块时最容易漏改、但最关键的文件之一。

## 9.5 `moduleControlRegistry.ts`

这是运行时模块执行中心：

- 模块注册
- 模块反注册
- 获取当前页模块上下文
- 执行 `tab/select` 切换
- 等待目标模块完成注册

## 9.6 `voiceControlState.ts`

这是解决“页面实例”和“弹窗实例”状态不同步问题的桥梁。

没有它，就会出现：

- 页面主视图切了 tab
- 但弹窗中的同模块依然停在旧 tab

## 10. 当前实现与旧 skill 文档的主要差异

下面这些点，是当前真实代码里已经存在、但旧 skill 文档不应再忽略的内容：

1. 当前不仅有“页面适配器 + 模块注册”，还额外有 **静态模块定义中心**。
2. 当前 parser 已支持 **clarify 澄清**，不只是普通命令解析。
3. 当前有 **pendingCommandStore**，支持跨页面跳转后续跑命令。
4. 当前有 **voiceControlStore**，专门解决弹窗双实例同步问题。
5. 当前支持 **debug 模式** 和 **backend 模式** 两种入口。
6. 当前 ASR 识别默认已经走 `CloudRecognizer`，而不是旧文档里强调的浏览器原生识别。
7. 当前项目已经把 **DataLoom 文档纳入仓库**，但尚未真正接线。
8. 当前模块接入并不是简单在组件里手写 register，实际更推荐通过 `module-definitions.ts + useHydrateContext()` 的方式接入。

## 11. 后续新增模块怎么接入

这里说的是“新增一个现有页面中的语音可控模块”，不是新增页面。

## 11.1 标准接入步骤

### 步骤 1：先在静态模块定义里补充模块语义

修改：

- `src/voice/core/module-definitions.ts`

需要补：

- `moduleId`
- `title`
- `aliases`
- `controls`
- 每个 control 的 `controlId`
- 每个 control 的 `controlType`
- 静态 `options`

这一步的意义：

- 让 parser 在目标页面未挂载时也能理解这类指令
- 让跨页面语音跳转时依然可以识别目标模块

### 步骤 2：在实际业务组件里注册运行时能力

根据页面类型，优先复用已有 hook：

- 整体看板类模块：`src/views/opcenter/hooks/useHydrateContext.ts`
- 规模看板类模块：`src/views/mktdash/hooks/useHydrateContext.ts`

如果现有 hook 已能满足控件类型：

- 直接在业务组件中调用 hook
- 传入 `moduleId`
- 传入当前 tab/select 的响应式状态

如果现有 hook 不够：

- 可以扩展 hook 支持更多控件
- 或者手动调用 `moduleControlRegistry.register()`

### 步骤 3：保证模块 ID 命名一致

命名规则要统一：

- 静态定义中的 `moduleId`
- 组件注册时的 `moduleId`
- 语音动作里的 `moduleId`
- 弹窗组件同步时使用的 `modalComponentId`

当前项目中常见规则：

- 页面主模块：`opcenter:xxx` / `mktdash:xxx`
- 弹窗组件 ID：把 `:` 转成 `.`

### 步骤 4：如果模块支持弹窗，确认弹窗实例也会同步

如果该模块存在“页面视图 + 弹窗视图”双实例：

- 执行语音切换后，需要通过 `voiceControlStore.publish()`
- 弹窗端通过 watch 同步

如果复用已有 `useHydrateContext()`，通常这部分已经帮你处理了。

### 步骤 5：补充 aliases

模块接入成败很大程度取决于别名设计。

建议至少覆盖：

- 模块正式标题
- 业务常用简称
- 用户口语叫法
- 上下位称呼

例如当前项目就大量使用：

- 集团 / 总部 / 公司整体 / 上方
- 下属公司 / 下属物业公司 / 区域公司 / 下属 / 下方

### 步骤 6：补充控件 options 的 aliases

如果控件选项也有口语化说法，记得给 option 配 `aliases`。

否则 parser 很可能只能识别最正式的展示文案。

## 12. 新增模块时涉及哪些文件

按“普通新增模块”的最常见情况，通常会涉及这些文件：

### 必改

- `src/voice/core/module-definitions.ts`
- 对应业务组件 `.vue`

### 视情况修改

- `src/views/opcenter/hooks/useHydrateContext.ts`
- `src/views/mktdash/hooks/useHydrateContext.ts`
- `src/voice/types/module.ts`
- `src/voice/registry/moduleControlRegistry.ts`
- `src/voice/core/parser.ts`

### 一般不用改

- `src/voice/core/orchestrator.ts`
- `src/voice/core/dispatcher.ts`
- `src/voice/registry/executionRules.ts`

因为现有模块动作类型已经覆盖了：

- 打开模块
- 模块 tab 切换
- 模块 select 切换

## 13. 如果不是新增模块，而是新增页面/新筛选项

## 13.1 新增页面

通常需要改：

- `src/voice/types/command.ts`
- `src/voice/core/dictionary.ts`
- `src/voice/core/parser.ts`
- `src/voice/adapters/新页面Adapter.ts`
- `src/voice/registry/executionRules.ts`
- 新页面入口 `.vue`

核心原因：

- PageId 要扩展
- 页面别名要能识别
- parser 要知道页面怎么映射
- 页面要有适配器负责执行
- 页面要定义动作顺序
- 页面自身要注册/注销适配器

## 13.2 新增筛选项

通常需要改：

- `src/voice/types/command.ts`
- `src/voice/core/dictionary.ts`
- `src/voice/core/parser.ts`
- 对应页面 adapter
- 页面入口组件 expose
- `src/voice/registry/executionRules.ts`

例如以后若要支持：

- 季度
- 周期
- 城市级过滤
- 指标口径

都要按这条链路扩展。

## 14. 当前实现的限制与待办

## 14.1 仅支持两个 PageId

当前 `PageId` 只有：

- `home`
- `opcenter`

说明页面语音覆盖范围目前还不算全项目通用，只覆盖了首页和驾驶舱主页。

## 14.2 backend 分支仍未真正接后端

`useVoiceControl.ts` 中：

- `requestBackendIntent()` 仍返回 `null`

这意味着：

- backend 模式并不是真正的 AI 后端模式
- 只是语音输入通路与本地执行结合

## 14.3 DataLoom 文档与当前动作模型还未对齐

后端文档中出现的动作：

- `setStartYearPeriod`
- `setEndYearPeriod`

而当前前端只实现了：

- `setYearPeriod`

后续正式对接 DataLoom 时，这里必须做协议映射或动作扩展。

## 14.4 当前澄清结构与 DataLoom 协议不一致

当前本地澄清使用的是对象选项结构，包含：

- `label`
- `title`
- `aliases`

而 DataLoom 文档里的 `clarify.options` 当前是 `string[]`。

这意味着后续接后端时要调整：

- `clarifyContextStore`
- UI 展示
- follow-up 解析逻辑

## 14.5 当前面板还没有完整承接 DataLoom 的流式协议

DataLoom 文档里有：

- `stream`
- `action_plan`
- `clarify`
- `done`
- `error`
- `recommend`
- `chatReply`

当前 UI 面板还没完整对应这些消息类型。

## 15. 推荐的后续维护原则

1. 以后新增模块，先补静态定义，再补运行时注册。
2. 不要只改业务组件，不改 `module-definitions.ts`。
3. 不要只改 parser，不补 adapter 和 executionRules。
4. 不要把语音逻辑散落在业务组件里，优先通过 registry、adapter、hook 统一接入。
5. DataLoom 正式接入后，要把“本地解析模式”和“后端动作计划模式”区分清楚。
6. 新增口语别名时，优先从真实用户说法反推，而不是只用书面名称。
7. 弹窗与页面双实例模块，必须验证状态同步，不然最容易出现“主视图已切换，弹窗没切”的假成功。

## 16. 一句话总结

当前项目已经完成了一套 **可运行的前端语音控制基础设施**：

- 有入口
- 有识别
- 有解析
- 有编排
- 有页面适配器
- 有模块注册中心
- 有静态模块语义库
- 有弹窗状态同步

但 **后端 AI 动作计划链路还没有正式接入**。

因此，后续工作的正确方向不是推翻重做，而是：

> 在现有前端语音执行骨架上，继续补齐 DataLoom/后端 AI 指令通道，并同步更新项目文档与 skill 文档。
